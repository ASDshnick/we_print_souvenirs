Документ 1: BRD (Business Requirements Document)
   1. Введение
1.1 Назначение документа


Данный документ описывает бизнес-требования к развитию сайта 3D-моделей. Цель — зафиксировать согласованное видение проекта, определить цели, масштаб и ожидаемые результаты для дальнейшей разработки.
1.2 Целевая аудитория


* Владелец проекта (спонсор)  
* Руководитель отдела 3D-моделирования  
* Администратор сайта  
* Команда разработки (для понимания контекста)
2. Бизнес-требования
2.1 Проблема


* Существующая базовая версия сайта не позволяет принимать заказы онлайн.  
* Взаимодействие с клиентами происходит через Telegram и электронную почту напрямую.  
* 3D-дизайнеры тратят много времени на уточнение требований к моделям, что увеличивает издержки и замедляет выполнение заказов.  
* Клиенту необходимо знать контакт специалиста, чтобы сделать заказ, что снижает удобство.
2.2 Возможность


* Запуск полноценного каталога и формы заказа позволит выйти на новые рынки (например, туристический бизнес, производство сувениров).  
* Автоматизация сбора требований через форму заказа сократит время дизайнеров на уточнение.  
* Удобный интерфейс для заказа привлечёт клиентов, которые ценят простоту.
2.3 Качественные бизнес-цели 


* Создать централизованную платформу для заказа 3D-моделей и услуг 3D-печати.  
* Перевести все коммуникации с клиентами на сайт, сократив использование внешних мессенджеров.  
* Повысить удовлетворённость клиентов за счёт прозрачности статуса заказа.
2.4 Бизнес-задачи (SMART)


Специфичная
	Запустить новый функционал сайта: личный кабинет, форму заказа, примеры работ с 3D-просмотром
	Измеримая
	Сократить время, затрачиваемое дизайнерами на уточнение требований, на 50% за 2 месяца после запуска.
	Достижимая
	Ресурсы команды (дизайнеры, контент) и разработка в рамках выделенного времени позволяют достичь цели.
	Релевантная
	Соответствует стратегии цифровизации и расширения рынка.
	Временная
	Запуск к началу летнего сезона.
	2.5 Метрики


* Количество заказов, оформленных через форму на сайте (в месяц).  
* Снижение количества уточняющих вопросов в Telegram/почте.  
* Увеличение числа зарегистрированных пользователей.  
* Время от заказа до подтверждения (после внедрения автоматизации).

4. Обзор бизнес-процессов
4.1 Текущий процесс 
1. Клиент находит информацию о студии (через соцсети, рекомендации).  
2. Клиент связывается с дизайнером напрямую через Telegram или почту.  
3. Долгое уточнение требований (формат, полигонаж, сроки).  
4. Согласование цены и условий.  
5. Дизайнер выполняет модель, отправляет клиенту.  
6. Коммуникация по правкам происходит в том же канале.


Идет потеря времени на уточнения, неструктурированные требования, сложность отслеживания истории.
4.2 Целевой процесс


1. Клиент заходит на сайт, изучает примеры работ и услуги.  
2. Регистрируется в личном кабинете (или сразу заполняет форму заказа).  
3. Заполняет форму заказа: указывает тип модели, требования (полигоны, «важно»), сроки, если нужна печать — выбирает способ доставки, количество, цветность.  
4. Система создаёт заказ, уведомляет администратора.  
5. Администратор назначает дизайнера, заказ получает статус.  
6. Дизайнер и клиент общаются через встроенный чат на сайте (обсуждение деталей, правок).  
7. После выполнения модель загружается в личный кабинет клиента, приходит уведомление (бот).  
8. Клиент может скачать модель в любое время, а также дать согласие на размещение модели в примерах работ.

	Документ 2: SRS (Software Requirements Specification)


1. Введение
1.1 Назначение


Документ описывает функциональные и нефункциональные требования к веб-сайту для заказа 3D-моделей и услуг 3D-печати. Основа для проектирования, разработки и тестирования.
1.2 Область применения


Продукт — веб-сайт, доступный через браузер (ПК, мобильные устройства). Основные возможности: каталог/портфолио, форма заказа, личный кабинет, административная панель.
1.3 Аудитория


* Команда разработки (для реализации)  
* Тестировщики (для создания тест-кейсов)  
* Администраторы (для понимания функционала)
2. Общее описание
2.1 Перспектива продукта


Продукт является развитием существующей базовой версии сайта. Вводится функционал приёма заказов онлайн, замена внешних каналов коммуникации на встроенный чат, автоматизация сбора требований.






2.2 Пользовательские классы


Гость
	Неавторизованный посетитель. Может просматривать примеры работ, изучать услуги, не может оформить заказ.
	Зарегистрированный пользователь
	Имеет личный кабинет, может оформлять заказы, отслеживать статус, общаться в чате, скачивать готовые модели.
	Администратор
	Управляет заказами, пользователями, примерами моделей, просматривает статистику.
	



2.3 Операционное окружение


* Браузеры: Google Chrome - приоритет; также поддержка Firefox, Safari, Edge.  
* Устройства: ПК, смартфоны (Android, iOS) — адаптивная вёрстка.  
* Хостинг: любой, обеспечивающий работу PHP/MySQL (конкретный не определён).  
* Хранение файлов: Google Диск (интеграция через API).


2.4 Ограничения


* Функция оплаты в первой версии не реализуется.  
* Возврат средств не предусмотрен.  
* Количество бесплатных правок — не более 3. При 6-й правке заказ блокируется для правок.  
* Срок разработки — к началу летнего сезона (условно: 31 мая).






2.5 Допущения


* Заказчики преимущественно используют мобильные устройства для оформления, поэтому формы адаптированы.  
* Пользователи согласны на регистрацию для оформления заказа.  
* Все файлы моделей для портфолио и готовых заказов будут загружены на Google Диск.  
* Яндекс.Метрика будет использована для сбора статистики.


**2.6 Зависимости**


* Наличие учётной записи Google API для доступа к Диску.  
* Опционально: настройка Telegram-бота для уведомлений.


3. Бизнес-правила


BR-01
	При оформлении заказа в личном кабинете клиент обязан указать требования к модели (полигоны, сроки указываются в форме чекбоксом ).
	BR-02
	Заказ требует дополнительной информации автоматически, если дизайнер не отреагировал в течение N часов.
	BR-03
	После готовности модели клиент получает уведомление (в личном кабинете, опционально — в Telegram/почте).
	BR-04
	Бесплатные правки — не более 3 итераций. При 4-й правке взимается дополнительная плата (на усмотрение администратора). При 6-й правке заказ блокируется для дальнейших изменений.
	BR-05
	Модель сохраняется в личном кабинете клиента бессрочно. Файлы на Google Диске хранятся ~30 дней, затем ссылка может быть обновлена администратором по запросу.
	BR-06
	Клиент может дать согласие на размещение выполненной модели в разделе «Примеры работ».
	







4. Функциональные требования


4.1 Примеры работ 


FR-01
	Система должна отображать список примеров работ (моделей) с превью и кратким описанием.
	FR-02
	При нажатии на карточку модели открывается страница с детальным 3D-просмотром.
	FR-03
	Система должна учитывать просмотр модели и 3D-просмотр как событие для статистики.
	FR-04
	Доступен поиск по примерам работ (по названию, тегам).
	FR-05
	Администратор может добавлять, редактировать и удалять примеры работ, загружать 3D-модель для просмотра (форматы fbx, OBJ и др.).
	

4.2 Форма заказа


FR-06
	Форма заказа доступна только авторизованным пользователям.
	FR-07.1
	Форма должна содержать поля: тип заказа (3D-модель / 3D-печать), требования, контактные данные (текстовые поля), Пользователь может отметить приоритетные параметры заказа: количество полигонов, срок выполнения.
	FR-07.2
	При выборе чекбокс приоритетного параметра пользователь опционально может указать в текстовом поле рядом с чекбоксом желаемые сроки и число полигонов.
	FR-08
	При выборе услуги 3D-печати отображаются дополнительные поля: способ доставки, количество экземпляров, цветная/одноцветная печать.
	FR-9[b]
	После отправки формы создаётся заказ со статусом «Новый».
	

**4.3 Личный кабинет


FR-10
	Пользователь может просматривать историю своих заказов (бессрочно).
	FR-11
	Для каждого заказа отображается: статус (название, процент выполнения, метки готовности), возможность скачать готовую модель (если статус «Готов»), ссылка на чат.
	FR-12
	В личном кабинете реализован мини-чат с дизайнером по каждому заказу. Сообщения сохраняются.
	FR-13
	Пользователь может скачать модель неограниченное количество раз, пока файл доступен на Google Диске.
	FR-14
	Пользователь может запросить обновление ссылки на скачивание, если срок хранения истёк.
	

**4.4 Административная панель**


FR-15
	Администратор имеет доступ к списку всех заказов с фильтрацией по статусу, дате, пользователю.
	FR-16
	Администратор может изменять статус заказа, устанавливать процент готовности, добавлять метки (например, «ожидает оплаты правок», «готов к отгрузке»).
	FR-17
	Администратор может просматривать статистику просмотров моделей в разделе «Примеры работ».
	FR-18
	Администратор может управлять пользователями (блокировка, сброс пароля, назначение роли).
	FR-19
	Администратор может настроить автоматическое подтверждение заказа (время ожидания реакции дизайнера).
	FR-20
	При смене статуса заказа на «Готов» система должна отправить уведомление пользователю (в личный кабинет и, опционально, через внешний бот).
	4.5 Регистрация и аутентификация


FR-21
	Регистрация по email и паролю.
	FR-22
	Если незарегистрированный пользователь пытается оформить заказ, система предлагает зарегистрироваться и перенаправляет обратно.

5.3 Обслуживание данных


* История заказов хранится бессрочно в БД.  
* Файлы моделей (результаты заказов) хранятся на Google Диске не менее 30 дней. По истечении срока администратор может продлить ссылку или обновить её.  
* Данные пользователей хранятся в соответствии с законодательством (без передачи третьим лицам).
6. Внешние требования
6.1 Пользовательский интерфейс


* Адаптивный дизайн под мобильные устройства и ПК.  
* Для 3D-просмотра используется встраиваемый просмотрщик
* Интерфейс личного кабинета — минималистичный, акцент на список заказов и чат.


6.2 Программные интерфейсы


* Google Диск API — для загрузки файлов моделей и получения публичных ссылок.  
* Яндекс.Метрика — для сбора аналитики (посещаемость, просмотры страниц).  
* Telegram Bot API (опционально) — для отправки уведомлений о статусе заказа.


6.3 Коммуникационные интерфейсы


* В системе реализуется встроенный чат, привязанный к конкретному заказу.
* Обмен сообщениями осуществляется между пользователем и администратором/дизайнером.
* Сообщения сохраняются в БД.
* Новые сообщения отображаются в интерфейсе без перезагрузки страницы или с автоматическим обновлением.
* Уведомления приходят в личный кабинет и, при настройке, в Telegram/почту.
7. Нефункциональные требования
7.1 Производительность


* Время загрузки страниц не более 5 секунд при стандартном интернет-соединении.  
* 3D-просмотр модели должен инициироваться не более чем за 10 секунды после нажатия.
7.2 Надежность и доступность


* Доступность сайта — 24/7, за исключением плановых технических работ (предупреждение администраторов).

8.1 Глоссарий




3D-просмотр
	Интерактивный просмотр модели в браузере с возможностью вращения, масштабирования и изменения ракурса.
	Заказ
	Запрос пользователя на разработку 3D-модели или 3D-печать.
	Статус заказа
	Текущее состояние заказа в жизненном цикле обработки.
	Правка
	Итерация изменения модели по замечаниям клиента.
	Личный кабинет
	Раздел сайта, доступный авторизованному пользователю для управления заказами и сообщениями.
	Администратор
	Пользователь с расширенными правами управления системой.
	Дизайнер
	Исполнитель, который обрабатывает заказ и вносит изменения в модель.
	Встроенный чат
	Канал переписки внутри системы, привязанный к конкретному заказу.
	Метка
	Дополнительный атрибут заказа, используемый для уточнения состояния или приоритета.
	Превью
	Изображение или упрощённое представление модели в каталоге.
	Файл модели
	Итоговый или исходный 3D-файл, прикреплённый к заказу.
	Облачное хранилище
	Интерфейс для загрузки, хранения и получения файлов во внешнем хранилище.
